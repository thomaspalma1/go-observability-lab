A **Go** study lab focused entirely on **observability**. The API itself (a **"load runner"** that fires load against a target endpoint) has an intentionally simple domain; the real goal of this project is to serve as a controlled environment for exploring **structured logging**, **metrics**, **distributed tracing**, **health checks**, **SLOs**, and **profiling**, all running in a local stack via **Docker Compose**.

## **Architecture**

The service has two internal responsibilities:

- **Load Runner**: fires HTTP requests against a configurable URL, at a configurable rate (`requests_per_second`) and duration, aggregating the results (**success**, **failure**, **latency**).
- **Target**: a simple endpoint (`/target/ping`) that serves as the target for load tests, with support for artificial delay via query param, used to simulate slowness and test **SLOs**.

| Endpoint | Purpose |
|----------|---------|
| `/load-test/run` | Fires load (**goroutines + HTTP client**) |
| `/load-test/{id}/results` | Returns the results of a load test |
| `/target/ping` | Load test target (**optional `delay_ms`**) |
| `/healthz`, `/readyz` | Liveness and readiness endpoints |
| `/metrics` | Exposes **Prometheus** metrics |
| `/debug/pprof/*` | Native **Go** profiling endpoints |
| `/swagger/*` | Interactive **Swagger** documentation |

## **Observability stack**

| Tool | Role |
|--------------|-----------------------------------------------------|
| **Prometheus** | Collects and stores metrics, evaluates **SLO/alert rules** |
| **Grafana** | Visualization (**auto-provisioned dashboards**) |
| **Jaeger** | Stores and displays **distributed traces** (via **OTLP**) |
| **Alertmanager** | Receives and manages alerts fired by **Prometheus** |

## **Observability pillars implemented**

### **1. Structured logging**
Every request is logged as **JSON** via `log/slog`, including a **`request_id`** that correlates the whole request — generated by middleware and propagated in the **`X-Request-ID`** header.

### **2. Metrics (Prometheus)**
- **`http_requests_total`**: request counter, by method/route/status
- **`http_request_duration_seconds`**: latency histogram, by method/route
- Native **Go runtime metrics** (`go_goroutines`, `go_memstats_*`, `process_cpu_seconds_total`) exposed automatically

### **3. Distributed tracing (OpenTelemetry + Jaeger)**
- Automatic instrumentation of every HTTP request via **`otelgin`**
- Manual spans in the load runner (**`loadtest.run`** → **`loadtest.fire_request`**), propagating context between the original HTTP request and the background firing goroutines (via **`context.WithoutCancel`**)
- Errors and status of each fired request are recorded on the corresponding **span**

### **4. Health checks**
- **`/healthz`**: simple **liveness check**
- **`/readyz`**: real **readiness check**, reflecting the number of concurrent load tests currently running in the process

### **5. SLO, Error Budget and Alerts**
- **SLO defined**: **95%** of requests to **`/target/ping`** must respond within **50ms**
- **Prometheus recording rules** calculate the **violation ratio** and **error budget burn rate**
- An alert (**`TargetPingSLOBurnRateHigh`**) fires when the **burn rate** exceeds **10x** the sustainable rate, visible in both **Prometheus** and **Alertmanager**

### **6. Profiling (pprof)**
Native Go endpoints (**`/debug/pprof/*`**) exposed through **Gin**, allowing **CPU**, **heap**, and **goroutine profiles** to be captured during real load tests.

## **Usage example**

Fire a load test:

```yaml
curl -X POST http://localhost:8082/load-test/run \
  -H "Content-Type: application/json" \
  -d '{"target_url": "http://api:8082/target/ping", "requests_per_second": 50, "duration_seconds": 30}'
```

Check the results:

```yaml
curl http://localhost:8082/load-test/{test_id}/results
```

Simulate slowness (to test the **SLO**):

```bash
curl -X POST http://localhost:8082/load-test/run \
  -H "Content-Type: application/json" \
  -d '{"target_url": "http://api:8082/target/ping?delay_ms=200", "requests_per_second": 10, "duration_seconds": 90}'
```

## **Profiling findings**

A **CPU profile** captured during a sustained load test (**200 req/s**) showed that over **50%** of CPU time was spent in **`Syscall6`** (a generic Linux syscall used internally for **network I/O**), rather than in application code. This confirms that, under this workload, the **load runner's bottleneck** is **network I/O**, not CPU processing in the Go application itself.

## **Project structure**

├── internal/

## **Tech stack**

- **Go** with **Gin**
- **Prometheus** (`client_golang`) for **metrics**
- **OpenTelemetry** for **tracing** (exported via **OTLP/gRPC**)
- **swaggo** for **OpenAPI/Swagger** documentation
- **Docker Compose** orchestrating the whole stack locally
